# 반려동물 비전 AI 파이프라인 계획 (안구 질환)

## 1. 서버 환경 평가
- 현재 작업 디렉토리: `/root/medical_AI/hkh/jacob/ktb_fp_vetChatbot`
- 네트워크 스토리지 마운트 포인트: `/root/medical_AI` (무려 100TB 가용 용량 확인)
- 결론: 201GB 원본 안구 질환 데이터셋을 문제없이 다운로드하고 압축 해제할 수 있는 최적의 환경!

## 2. 향후 진행 스텝 제안
1. `download_aihub.sh` 스크립트를 서버(`/root/medical_AI/hkh/jacob/ktb_fp_vetChatbot`)에 업로드 (혹은 클론)
2. `download_aihub.sh` 를 사용하여 4번 안구 질환 데이터셋의 원본을 전체 다운로드 (`aihubshell` 사용)
3. 다운로드 완료 후, 압축 해제 및 디렉토리 구조 최적화 스크립트 작성 (`preprocess_vision.py`)
4. ResNet 또는 EfficientNet 기반의 이미지 분류 모델 학습 스크립트 짜기 (`train_vision_eye.py`)
## 3. 데이터셋별 활용 가치 평가 및 분류 모델 파이프라인
* 3번 데이터 (건강정보): 비만도/체형 분류 (우선순위: 낮음)
* 4번 데이터 (안구질환): 결막염, 백내장 등 직관적인 질병 예측 (우선순위: **가장 높음**)
* 5번 데이터 (피부질환): 뾰루지, 감염 등 광범위한 진단 (우선순위: 높음)

각 데이터셋은 완전히 독립된 목적으로 수집되었으므로, 하나의 통합 모델에 욱여넣는 것보다 **각각의 전문 분류 모델(Expert Models)을 개발하고(앙상블 혹은 라우팅) LLM이 이를 관장하게 하는 MOE(Mixture of Experts)와 유사한 아키텍처**를 구성하는 것이 압도적으로 성능이 좋습니다.
## 4. 서버 배포 및 아키텍처 (멀티 모델 서빙)
챗봇 서버에 4개의 모델(LLM + 3개의 비전 모델)을 한꺼번에 띄우는 것은 리소스 측면에서 비효율적일 수 있습니다.

**[추천 아키텍처]**
- **메인 챗봇 서버 (LLM Server):** 사용자와 직접 통신하며, LLaMA(1번 데이터 파인튜닝) 모델이 상주합니다. (RAG VectorDB 연결)
- **비전 분석 API 서버 (CV Microservices):** 안구(4번), 피부(5번) 분류 모델은 매우 가볍기 때문에 (수십~수백 MB 수준), 하나의 작고 저렴한 GPU 서버(또는 CPU 환경에서도 구동 가능)에 묶어서 별도의 API(FastAPI 등)로 띄웁니다.
  - 사용자가 이미지를 전송 -> 메인 서버가 이미지를 받아 비전 API 서버로 전달 -> 피부/안구 중 확률이 높은 분석 텍스트를 돌려받음 -> LLM이 이를 조합하여 답변 생성.
- **CPU 서버 인프라 (비전 다중 모델):**
  - EfficientNet, ResNet 기반의 3~5번 분류 모델들은 Inference(추론) 단계에서 CPU만으로도 불과 0.5~1.5초 내외의 빠른 처리 속도를 보여줍니다.
  - VRAM(그래픽 메모리) 한계의 구애를 받지 않고 서버의 메인 RAM(시스템 메모리)을 활용하여 여러 모델을 동시에 띄울 수 있다는 장점이 큽니다.
  - 결론: 챗봇(LLM)은 GPU 모델, 그 외 이미지 판독(CV) 모델들은 저렴한 고용량 메모리 기반의 CPU (또는 아주 작은 저사양 GPU 인스턴스)로 띄우는 것이 실무 표준이며 가장 합리적입니다!
## 5. 헬스케어 비디오 분석 결과 출력 파이프라인 (2-Step 접근법)
- **Phase 1 (MVP 배포 - 규칙 기반):** 보행 영상 모델의 출력값(Score)과 하드코딩된 규칙 기반(Rule-based) 설명을 사용자에게 그대로 출력.
- **Phase 2 (고도화 단계 - LLM 결합):** Rule-based 텍스트 대신, 백엔드에서 Score 데이터를 LLM 시스템 프롬프트로 전달하여 LLM이 자연스러운 수의사 말투로 분석 리포트를 동적 생성.
- **아키텍처 변경점:** 클라이언트 UI/단말기 구조는 전혀 바꿀 필요가 없으며, 백엔드 라우터(Controller)에서 결과 텍스트를 생성하는 함수(Function)만 Rule-based에서 LLM API Call로 갈아끼우면 되는 아주 유연하고 안전한 구조.
## 6. vLLM 마이그레이션 가능성과 AI-Orchestrator 통합 아키텍처
현재 MVP 모델 (FastAPI 기반 일반 추론 -> AI-Orchestrator 연결)에서 향후 고도화 시 vLLM 엔진으로 교체하는 작업은 플랫폼 측면에서 **전혀 어렵지 않습니다.**

### 기존(MVP) 방식의 흐름
1. `AI-Orchestrator (FastAPI Gateway)`가 사용자 요청 접수
2. Orchestrator가 `vetChatbot의 FastAPI 엔드포인트`로 JSON 요청 전달
3. `vetChatbot FastAPI` 라우터가 Pytorch/Transformers의 `model.generate()` 함수를 실행하여 아주 느린 속도로 텍스트를 한 글자씩 생성 후 회신
4. Orchestrator가 결과를 모아서 백엔드(앱단)로 전송

### vLLM 마이그레이션 시 (고도화 단계)
1. 내부의 파이썬 코드를 뜯어고칠 필요 없이, 아예 챗봇용 FastAPI 컨테이너와 `model.generate()` 로직을 지워버립니다.
2. 터미널에서 `python -m vllm.entrypoints.openai.api_server --model [우리가 훈련한 LoRA 병합 폴더]` 명령어 딱 한 줄만 실행합니다.
   * vLLM이 알아서 모델을 메모리에 최적화해서 올리고, OpenAI와 100% 동일한 고속의 API 주소(`http://localhost:8000/v1/chat/completions`)를 자동으로 뚝딱 열어줍니다.
3. `AI-Orchestrator`에서는 모델을 호출하는 주소(URL)만 방금 열린 vLLM 주소로 수정하면 끝입니다!

결론: 현재 구축해두신 `AI-Orchestrator (API Gateway)` 구조 자체가 마이크로서비스 확장에 완벽하게 최적화된 아키텍처입니다. 나중에 엔진을 Hf -> vLLM으로 바꾸는 것은 코드 수정이 아니라 단순히 "구동 프로그램을 바꾸고 주소만 연결하는 수준"이므로 추후 전환 비용은 '제로'에 가깝습니다.

## 7. 자체 Qwen 파인튜닝 모델 벤치마크 결과 해석 및 결정 (25.10.x)
- **Qwen-7B-LoRA**:
  - 속도: **20~25 TPS** (매우 빠름, 실시간 채팅 적합)
  - VRAM: **6.4GB** (가장 가벼움)
  - 답변 품질: 평이함. "위험합니다. 병원 가세요." 같은 기계적이고 뻔한 답변.

- **Qwen-14B-LoRA (Base)**:
  - 속도: **~14 TPS** (보통)
  - VRAM: **19.1GB** (가장 크고 부담됨, 24GB GPU 거의 꽉 참)
  - 답변 품질: 7B보다 약간 낫지만, 여전히 기계적인 경고 위주.

- **Qwen-14B-Quantized (8-bit 양자화)**:
  - 속도: **~7.4 TPS** (가장 느림, 읽기에는 문제없는 수준)
  - VRAM: **15.5GB** (14B임에도 꽤 가벼움)
  - 답변 품질: **가장 우수함 (질문 의도 완벽 파악)**
    - "거품토에 하루 굶기는 민간요법"에 대해 -> "단기간의 급식(굶김)은 필요할 수 있으나 의사 지시에 따르라"고 정확히 조언.
    - "초콜릿에 억지로 소금물 먹이기"에 대해 -> "집에서 소금물이나 다른 약물을 강제로 먹이는 것은 위험하다"며 민간요법의 위험성을 콕 찝어 경고함 (가장 중요한 수의학적 포인트).

### MVP 배포용 최종 권장 모델: `Qwen-14B-Quantized`
* 이유: 수의사 챗봇의 핵심은 "위험한 민간 뇌피셜(소금물 등)을 막아주고 전문가의 관점"을 설명해주는 것인데, 양자화 모델만이 이 의도를 정확히 파악했습니다. TPS가 7.4 정도로 조금 느리긴 하지만, Streaming 방식을 사용하면 사용자가 답답함을 전혀 느끼지 않습니다.
### [심화 테스트] 7B LoRA vs 14B LoRA vs 14B Quantized (Base) 분석
신규 추가된 2가지 질문 (슬개골 탈구 예측, 종합백신/항체검사 논란)에 대한 테스트 결과.

**분석:**
1. **7B LoRA (약 24.3 TPS, 6.3GB)**
   - 매우 빠르고 가벼움.
   - 답변: 안타깝게도 "수컷 말티즈"임에도 "중성화 안된 암컷" 등 엉뚱한 환각(Hallucination)이 섞여 나오고, 거품토(위장관)나 종합백신에 대해 교과서적이지만 다소 뜬구름 잡는 답변 도출.
2. **14B LoRA (약 14.7 TPS, 19.1GB)**
   - 환각 증상은 7B보다 덜함 (3~7세 암컷이 흔하다는 통계적 사실로 우회 답변).
   - 여전히 사용자의 의도나 "어떻게 대처해야 할지"에 대한 공감형/지시형 답변보다는 "병원에 가라"는 말로 황급히 마무리하는 경향이 강함.
3. **14B Quantized Base (약 7.3 TPS, 15.5GB)**
   - 속도는 느리지만 **상담의 질이 압도적**임.
   - 슬개골 탈구 질문에 대해: 무작정 병원에 가라는 게 아니라 "관절염, 근육 손상일 수도 있다"며 감별 진단(DDx) 목록을 제공하고, **"병원에 가기 전까지 강제적으로 움직이지 않게 하고 편안한 환경을 제공하라"**는 명확한 '응급 처치(First aid)' 지침을 내려줌. (실제 수의사처럼 답변)

**🔥 최종 결론 및 다음 스텝:**
14B Quantized(순정) 모델이 가진 거대한 파라미터(지능) 자체가 '수의학적 추론력'에서 7B를 압도하고 있습니다. 
따라서 이 똑똑한 14B 모델 베이스에 우리가 만든 수의사 파인튜닝 지식(LoRA)을 융합(`merge_lora.py`)하고 다시 양자화(`quantize_8bit-for-merged.py`)하는 작업은 **최강의 퍼포먼스를 내기 위한 가장 완벽한 선택**이 될 것입니다!

### (참고) 클라우드 GPU와 HuggingFace 업로드/다운로드 속도 이유
H100과 같은 최고사양 GPU를 사용하더라도, HuggingFace Hub로 수십 기가바이트(GB)의 모델을 업로드하는 속도가 느린 이유는 **GPU의 연산 성능과는 무관한 네트워크/대역폭 문제**입니다.

1. **GPU와 네트워크는 별개**: H100은 '연산'을 빠르게 해줄 뿐, 인터넷(랜선) 속도를 올려주지는 않습니다. 업로드 속도는 그 서버가 달려있는 데이터센터(AWS, RunPod 등)의 **외부망 아웃바운드(Outbound) 대역폭**에 전적으로 의존합니다.
2. **해외망(HuggingFace 서버) 속도 제한**: HuggingFace의 메인 저장소(Storage 서버)는 주로 미국이나 유럽(AWS US-East 등)에 위치해 있습니다. 만약 지금 빌려 쓰시는 서버가 한국(Seoul) 리전이라면, 한국에서 미국 스토리지까지 태평양 해저 케이블을 건너서 수십 GB의 파일 단위(safetensors 등)를 쏘는 과정에서 어쩔 수 없는 물리적인 지연시간(Latency)과 대역폭 병목이 발생합니다.
3. **업로드 방식의 특성**: 다운로드는 CDN(Content Delivery Network) 캐시를 통해 전 세계 어디든 빠르게 쏴주지만, 무언가를 클라우드 원본 저장소에 "업로드(Write)" 할 때는 무결성 검증, 분할 전송(Multipart Upload), 해시 연산 등이 엄격하게 들어갑니다. (특히 Git LFS 기반 시스템을 기반으로 한 HF Api 특성 상 더 느리게 체감됩니다.)
- **AutoAWQ 양자화 에러 대응 (Transformers 버전 충돌)**
  - AutoAWQ 0.2.x 버전은 `transformers>=4.45.0`을 요구하지만, 해당 버전부터 14B 모델의 Attention layer 맵핑(`causal_mask_mapping`) 구조가 변경되어 AWQ의 Calibrator와 런타임 에러(`AttributeError: 'Catcher' object has no attribute 'attention_type'`)가 발생합니다.
  - **해결책 1**: 8-bit 양자화(BitsAndBytes)를 사용하여 에러 원천 차단. (VQA를 위한 말투 퀄리티 측면에서도 8-bit가 4-bit(AWQ)보다 훨씬 우수함이 여러 테스트결과 입증되었습니다.)
  - **해결책 2 (굳이 AWQ를 고집할 경우)**: AutoAWQ 소스 코드를 해킹하든가 호환 패치를 기다려야 하므로 권장하지 않습니다. 현재 `trl` 패키지의 요구 버전 충돌까지 고려하면 8-bit 양자화가 가장 정신건강에 좋습니다.
### [Troubleshooting] Qwen2.5-14B AWQ 양자화 종속성 에러
- **원인:** `AutoAWQ` 최신 버전(0.2.x)이 `transformers` 코드를 참조할 때, 존재하지 않는 `qwen3` 모듈이나 최신 어텐션 맵핑 방식을 강제 요구하면서 버전 꼬임 발생.
- **해결책 (vLLM 공식 권장 방식):** AutoAWQ 제작자가 공식적으로 권장한 대로 `llmcompressor` 라이브러리로 마이그레이션 해야 합니다. `llmcompressor`는 최신 `transformers`와 100% 호환되며, vLLM 팀에서 직접 관리하는 공식 4-bit 양자화 도구(GPTQ/AWQ 지원)입니다.
### [참조] 8-Bit (BnB) vs 4-Bit (AWQ/GPTQ) 양자화 속도 차이 원인
1. **8-Bit (BitsAndBytes):** **선형 압축(Linear Quantization)** 방식. 단순히 가중치 숫자를 절반 크기(FP16 -> INT8)로 잘라내기만 합니다. (예: 1.2345 -> 1.2). Calibration(보정) 과정 없이 즉시 깎아버리므로 초 단위로 끝납니다.
2. **4-Bit (GPTQ/AWQ):** **보정(Calibration) 기반 비선형 압축** 방식. 단순히 4-Bit로 잘라내면 모델이 완벽한 바보가 되므로, 수백~수천 개의 문장(샘플)을 모델에 직접 밀어 넣고 추론시켜봅니다. 각 층(Layer)마다 "이 부분(가중치)은 답변할 때 중요하네? 여긴 조금 덜 중요하네?"를 수학적으로 계산해가면서(Hessian Matrix 계산 등) **영리하게 깎아내는 복잡한 최적화 과정**이 들어갑니다. 그렇기에 수 분~수십 분이 소요되는 것이 정상입니다.

### [참조] GPTQ(llmcompressor) 양자화 로그의 `error` 수치 의미
로그에 표시되는 `METRIC - error 9143.54` 등은 **프로그램이 고장 난 오류(Bug)가 아닙니다.** 이것은 16-bit 숫자를 4-bit로 강제로 우겨넣으면서 발생한 **수학적 오차(Quantization Error / MSE Loss)**를 측정한 지표입니다. 각 레이어(층)마다 원래의 똑똑한 가중치 대비 얼마나 손실이 발생했는지 기록해 주는 정상적인 성능 모니터링 로그입니다.

## 8. 챗봇 서버(LLM) 배포 스펙 가이드: Qwen-14B (8-bit 양자화 모델 기준)
14B(140억 개 파라미터) 사이즈의 거대한 모델을 8-bit로 양자화(압축)하여 서비스 서버(Production)에 띄우실 때 필요한 H/W 권장 스펙입니다.

- **VRAM (그래픽 메모리) 요구량:**
  - **순수 모델 적재 사이즈**: 약 14.5GB ~ 15.5GB
  - **맥락(Context) 및 생성(KV Cache) 여유 공간**: 최소 2~4GB 추가 필요 (사용자가 긴 질문을 하거나 긴 문서를 찾아서 RAG로 던져줄 때 필요함)
  - ❗️ **최소 권장 VRAM**: **20GB 이상** (24GB VRAM을 가진 카드가 가장 이상적입니다.)

- **추천 가성비 GPU (Cloud / RunPod 기준):**
  - ⭐️ **NVIDIA RTX 3090 / RTX 4090 (24GB)**: 가장 저렴하면서도 초당 20~30단어를 뽑아내는 속도를 보장합니다. (가장 추천)
  - **NVIDIA A10G (24GB) / L4 (24GB)**: 클라우드(AWS 등)에서 엔터프라이즈 용으로 띄울 때 가장 무난하고 안정적인 서버용 그래픽카드입니다.
  - **NVIDIA RTX A5000 (24GB)**: 렌더링/AI 겸용 카드. 4090보단 느리지만 가격 대비 매우 훌륭합니다.

*(주의: 클라우드에서 서버를 임대할 때 '시스템 RAM(메인 메모리)'과 GPU의 'VRAM'은 다릅니다! 반드시 그래픽카드의 메모리(VRAM)가 24GB인지 확인해야 합니다. 시스템 메인 RAM은 32GB 이상이면 충분합니다.)*
