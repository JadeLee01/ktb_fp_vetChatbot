## [Insight] 8-bit 양자화 모델 응급처치의 한계 (실험 결과)
1. 무한 루프(치매 증상) 버그를 막기 위해 `repetition_penalty=1.2`와 `no_repeat_ngram_size=3`을 강제로 주입했습니다.
2. 결과: 같은 단어를 반복하는 증상은 해결되었지만, 모델이 "이전에 했던 말을 피해가야 한다"는 압박감에 시달려 **말도 안 되는 어투(~~다만, 어쩌구 저쩌구, 중국어 등)**를 내뱉거나 **스스로 질문을 창조하는 부작용(탈선 현상)**이 터졌습니다.
3. 소요 시간도 60~70초대로 RAG 환경에서는 고객 서비스용으로 절대 사용할 수 없는 수준입니다.
4. **결론:** `Qwen2.5-14B` 혹은 그 이상의 모델을 RAG와 함께 24GB GPU 환경에서 양자화 상태로 구동하려면, 단순 파이썬 `transformers` 스크립트 기반의 응급처치로는 품질을 보장할 수 없습니다. 
   - 데모(MVP) 단계에서는 **[7B-LoRA (양자화 안 한 원본/병합 모델)]**을 사용하여 안정성과 품질을 확보해야 합니다.
   - 추후 고도화 단계에서 vLLM을 도입하여 14B 단위의 안정적인 서빙 환경을 구축하는 로드맵이 가장 합리적입니다.
## [Fix] 7B 16-bit LoRA MVP 교체 및 RAG 경로 수정
1. 테스트를 통해 8-bit 양자화 모델이 과도한 반복과 성능 저하 문제를 일으킴을 파악했습니다.
2. 해결책으로 7B-LoRA (16-bit) 모델을 단일 24GB VRAM 환경에 띄우도록 `test_7b_lora_rag.py`와 `run_chatbot_server.py`를 교체했습니다.
3. RAG 디렉토리가 `vet_qa_db`가 아닌 실제 환경에 맞게 `chroma_db`로 수정되었습니다. 

## [Success] 7B 16-bit LoRA + Chat Template 결합 성공!
- 반복 억제 페널티를 전부 제거하고 Qwen 전용 Chat Template 프롬프트를 적용한 결과, 무한 반복(치매) 버그가 **100% 완벽하게 소멸**되었습니다.
- **VRAM 사용량:** 7B 모델과 LoRA 어댑터를 합쳐서 단 6.7GB ~ 7.5GB 수준으로 VRAM을 극도로 적게 소모합니다! (목표했던 24GB 단일 GPU에 아주 여유롭게 올라갑니다.)
- **시간 소요량:** 4가지 질문 모두 **5초~10초** 이내로, 챗봇 서비스로 바로 출시(MVP)해도 전혀 무방한 극강의 가성비와 속도를 뽐냈습니다.
