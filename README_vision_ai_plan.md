# 반려동물 비전 AI 파이프라인 계획 (안구 질환)

## 1. 서버 환경 평가
- 현재 작업 디렉토리: `/root/medical_AI/hkh/jacob/ktb_fp_vetChatbot`
- 네트워크 스토리지 마운트 포인트: `/root/medical_AI` (무려 100TB 가용 용량 확인)
- 결론: 201GB 원본 안구 질환 데이터셋을 문제없이 다운로드하고 압축 해제할 수 있는 최적의 환경!

## 2. 향후 진행 스텝 제안
1. `download_aihub.sh` 스크립트를 서버(`/root/medical_AI/hkh/jacob/ktb_fp_vetChatbot`)에 업로드 (혹은 클론)
2. `download_aihub.sh` 를 사용하여 4번 안구 질환 데이터셋의 원본을 전체 다운로드 (`aihubshell` 사용)
3. 다운로드 완료 후, 압축 해제 및 디렉토리 구조 최적화 스크립트 작성 (`preprocess_vision.py`)
4. ResNet 또는 EfficientNet 기반의 이미지 분류 모델 학습 스크립트 짜기 (`train_vision_eye.py`)
## 3. 데이터셋별 활용 가치 평가 및 분류 모델 파이프라인
* 3번 데이터 (건강정보): 비만도/체형 분류 (우선순위: 낮음)
* 4번 데이터 (안구질환): 결막염, 백내장 등 직관적인 질병 예측 (우선순위: **가장 높음**)
* 5번 데이터 (피부질환): 뾰루지, 감염 등 광범위한 진단 (우선순위: 높음)

각 데이터셋은 완전히 독립된 목적으로 수집되었으므로, 하나의 통합 모델에 욱여넣는 것보다 **각각의 전문 분류 모델(Expert Models)을 개발하고(앙상블 혹은 라우팅) LLM이 이를 관장하게 하는 MOE(Mixture of Experts)와 유사한 아키텍처**를 구성하는 것이 압도적으로 성능이 좋습니다.
## 4. 서버 배포 및 아키텍처 (멀티 모델 서빙)
챗봇 서버에 4개의 모델(LLM + 3개의 비전 모델)을 한꺼번에 띄우는 것은 리소스 측면에서 비효율적일 수 있습니다.

**[추천 아키텍처]**
- **메인 챗봇 서버 (LLM Server):** 사용자와 직접 통신하며, LLaMA(1번 데이터 파인튜닝) 모델이 상주합니다. (RAG VectorDB 연결)
- **비전 분석 API 서버 (CV Microservices):** 안구(4번), 피부(5번) 분류 모델은 매우 가볍기 때문에 (수십~수백 MB 수준), 하나의 작고 저렴한 GPU 서버(또는 CPU 환경에서도 구동 가능)에 묶어서 별도의 API(FastAPI 등)로 띄웁니다.
  - 사용자가 이미지를 전송 -> 메인 서버가 이미지를 받아 비전 API 서버로 전달 -> 피부/안구 중 확률이 높은 분석 텍스트를 돌려받음 -> LLM이 이를 조합하여 답변 생성.
- **CPU 서버 인프라 (비전 다중 모델):**
  - EfficientNet, ResNet 기반의 3~5번 분류 모델들은 Inference(추론) 단계에서 CPU만으로도 불과 0.5~1.5초 내외의 빠른 처리 속도를 보여줍니다.
  - VRAM(그래픽 메모리) 한계의 구애를 받지 않고 서버의 메인 RAM(시스템 메모리)을 활용하여 여러 모델을 동시에 띄울 수 있다는 장점이 큽니다.
  - 결론: 챗봇(LLM)은 GPU 모델, 그 외 이미지 판독(CV) 모델들은 저렴한 고용량 메모리 기반의 CPU (또는 아주 작은 저사양 GPU 인스턴스)로 띄우는 것이 실무 표준이며 가장 합리적입니다!
## 5. 헬스케어 비디오 분석 결과 출력 파이프라인 (2-Step 접근법)
- **Phase 1 (MVP 배포 - 규칙 기반):** 보행 영상 모델의 출력값(Score)과 하드코딩된 규칙 기반(Rule-based) 설명을 사용자에게 그대로 출력.
- **Phase 2 (고도화 단계 - LLM 결합):** Rule-based 텍스트 대신, 백엔드에서 Score 데이터를 LLM 시스템 프롬프트로 전달하여 LLM이 자연스러운 수의사 말투로 분석 리포트를 동적 생성.
- **아키텍처 변경점:** 클라이언트 UI/단말기 구조는 전혀 바꿀 필요가 없으며, 백엔드 라우터(Controller)에서 결과 텍스트를 생성하는 함수(Function)만 Rule-based에서 LLM API Call로 갈아끼우면 되는 아주 유연하고 안전한 구조.
## 6. vLLM 마이그레이션 가능성과 AI-Orchestrator 통합 아키텍처
현재 MVP 모델 (FastAPI 기반 일반 추론 -> AI-Orchestrator 연결)에서 향후 고도화 시 vLLM 엔진으로 교체하는 작업은 플랫폼 측면에서 **전혀 어렵지 않습니다.**

### 기존(MVP) 방식의 흐름
1. `AI-Orchestrator (FastAPI Gateway)`가 사용자 요청 접수
2. Orchestrator가 `vetChatbot의 FastAPI 엔드포인트`로 JSON 요청 전달
3. `vetChatbot FastAPI` 라우터가 Pytorch/Transformers의 `model.generate()` 함수를 실행하여 아주 느린 속도로 텍스트를 한 글자씩 생성 후 회신
4. Orchestrator가 결과를 모아서 백엔드(앱단)로 전송

### vLLM 마이그레이션 시 (고도화 단계)
1. 내부의 파이썬 코드를 뜯어고칠 필요 없이, 아예 챗봇용 FastAPI 컨테이너와 `model.generate()` 로직을 지워버립니다.
2. 터미널에서 `python -m vllm.entrypoints.openai.api_server --model [우리가 훈련한 LoRA 병합 폴더]` 명령어 딱 한 줄만 실행합니다.
   * vLLM이 알아서 모델을 메모리에 최적화해서 올리고, OpenAI와 100% 동일한 고속의 API 주소(`http://localhost:8000/v1/chat/completions`)를 자동으로 뚝딱 열어줍니다.
3. `AI-Orchestrator`에서는 모델을 호출하는 주소(URL)만 방금 열린 vLLM 주소로 수정하면 끝입니다!

결론: 현재 구축해두신 `AI-Orchestrator (API Gateway)` 구조 자체가 마이크로서비스 확장에 완벽하게 최적화된 아키텍처입니다. 나중에 엔진을 Hf -> vLLM으로 바꾸는 것은 코드 수정이 아니라 단순히 "구동 프로그램을 바꾸고 주소만 연결하는 수준"이므로 추후 전환 비용은 '제로'에 가깝습니다.

## 7. 자체 Qwen 파인튜닝 모델 벤치마크 결과 해석 및 결정 (25.10.x)
- **Qwen-7B-LoRA**:
  - 속도: **20~25 TPS** (매우 빠름, 실시간 채팅 적합)
  - VRAM: **6.4GB** (가장 가벼움)
  - 답변 품질: 평이함. "위험합니다. 병원 가세요." 같은 기계적이고 뻔한 답변.

- **Qwen-14B-LoRA (Base)**:
  - 속도: **~14 TPS** (보통)
  - VRAM: **19.1GB** (가장 크고 부담됨, 24GB GPU 거의 꽉 참)
  - 답변 품질: 7B보다 약간 낫지만, 여전히 기계적인 경고 위주.

- **Qwen-14B-Quantized (8-bit 양자화)**:
  - 속도: **~7.4 TPS** (가장 느림, 읽기에는 문제없는 수준)
  - VRAM: **15.5GB** (14B임에도 꽤 가벼움)
  - 답변 품질: **가장 우수함 (질문 의도 완벽 파악)**
    - "거품토에 하루 굶기는 민간요법"에 대해 -> "단기간의 급식(굶김)은 필요할 수 있으나 의사 지시에 따르라"고 정확히 조언.
    - "초콜릿에 억지로 소금물 먹이기"에 대해 -> "집에서 소금물이나 다른 약물을 강제로 먹이는 것은 위험하다"며 민간요법의 위험성을 콕 찝어 경고함 (가장 중요한 수의학적 포인트).

### MVP 배포용 최종 권장 모델: `Qwen-14B-Quantized`
* 이유: 수의사 챗봇의 핵심은 "위험한 민간 뇌피셜(소금물 등)을 막아주고 전문가의 관점"을 설명해주는 것인데, 양자화 모델만이 이 의도를 정확히 파악했습니다. TPS가 7.4 정도로 조금 느리긴 하지만, Streaming 방식을 사용하면 사용자가 답답함을 전혀 느끼지 않습니다.
